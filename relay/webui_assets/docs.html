Using Relay

Relay is a local-first model router: it plans on your machine, runs each subtask on your local model or a cloud specialist, and synthesizes one answer. This page covers everything you can configure and do from the UI.

How Relay works

Every request goes through four phases:

  1. Plan — your local model breaks the request into subtasks, each tagged with capabilities (coding, reasoning, current_info, …).
  2. Route — policy decides local or cloud per subtask, and which cloud model fits its capabilities.
  3. Execute — independent subtasks run in parallel; dependent ones wait for their inputs.
  4. Synthesize — the local model assembles one final answer from all results.

The plan preview

With Review plan before running enabled (Settings → Chat), Relay shows the plan before executing. You can edit each subtask's title and prompt, remove subtasks, add new ones, and pick a specific model per subtask from the dropdown (Auto uses the predicted model). Refresh routes re-predicts routing after your edits; Run plan executes. Disable the preview to send requests straight through.

The interface

Sidebar

  • Chats — every conversation, newest first; the active one is highlighted. New chat starts fresh.
  • Search chats — the box under "Chats" filters the list by title as you type.
  • Rename a chat — hover a chat and click the pencil; Enter saves, Escape cancels.
  • Delete a chat — hover a chat and click the ×, then click sure? to confirm. Deletion is permanent (it removes the chat from relay.history.json); deleting the open chat returns you to a fresh one.
  • Export — downloads the current chat as a Markdown file.
  • Status rows — orange dot: your local model; blue dot: how many distinct cloud models are reachable (or "cloud off"); green dot: the active privacy mode.
  • Collapse — the chevron next to the brand shrinks the sidebar to an icon rail; your choice is remembered.

Composer controls

  • Web pill — toggles web search for current-info subtasks (green when on). Same setting as Settings → Web search.
  • Route pill — Auto / Local / Cloud, color-coded: Auto is white, Local is orange, Cloud is blue. The filled pill is the active mode; it applies to every request until you change it.
  • Send / Stop — while a request runs, Send becomes a red Stop square; whatever already streamed in is kept and saved.

Landing screen

The status chips under the headline are live: current local model, cloud pool size, web search state, and privacy mode. They update whenever you save Settings. The example cards are tagged with the route they'll likely take.

Changing the local model

  1. Pull or load the model in your runtime, e.g. ollama pull qwen2.5:14b.
  2. Click List models to fetch what the runtime has, pick one in the Model field (or type its name).
  3. Ping local to confirm it responds, then Save settings.
The local model does triple duty: planner, worker for local subtasks, and synthesizer. A stronger local model improves every answer, even when the heavy lifting happens in the cloud.

For image attachments you need a vision-capable local model (e.g. ollama pull llava, qwen2.5-vl, llama3.2-vision) — or cloud enabled with a valid key.

Ollama models tagged -cloud (e.g. gpt-oss:120b-cloud) are Ollama Cloud models: they go through your local Ollama but execute on ollama.com's servers. The trace still labels them local because they use your local runtime, but the answer badge will say "Via Ollama Cloud" instead of "never left your machine". For fully on-device runs, pick a model without the -cloud tag.

Relay checks for this automatically: typing or saving a -cloud model in Settings shows an amber warning under the Model field, the save confirmation repeats the heads-up, and the model chips in the sidebar and landing screen explain it in their tooltip. It's a warning, not a block — Ollama Cloud models are a legitimate way to run big models without local VRAM; Relay just won't label them as private.

Cloud models & the pool

Flip on Cloud fallback (Settings) and pick a provider. With OpenRouter, one API key (sk-or-v1-… from openrouter.ai/keys) unlocks many models, and Relay routes each cloud subtask to a different model based on its capabilities. With OpenAI-compatible, all cloud subtasks use the single configured model.

The capability → model map

Settings → Cloud fallback → Cloud model pool holds a JSON object mapping capabilities to OpenRouter model slugs. A subtask may declare several capabilities; the highest-priority one that has an entry wins. Anything unmapped uses the Default model. Priority order: 

high_stakes → reasoning → math → coding → large_context
→ sources → current_info → vision → creative → general

Example — specialize three models and let everything else use the default:

{
  "coding":       "anthropic/claude-opus-4.8",
  "reasoning":    "anthropic/claude-opus-4.8",
  "current_info": "openai/gpt-5.5",
  "vision":       "google/gemini-2.5-pro"
}

Use any slug from openrouter.ai/models. To remove a model's role, delete its line and save. An empty map {} pins every cloud subtask to the default model. Invalid JSON is rejected safely — it never wipes your pool.

The sidebar status line shows how many distinct cloud models are reachable (e.g. 3 cloud) — that's the pool plus the default, de-duplicated.

Routing controls

The Route pill (Auto / Local / Cloud)

Above the message box. It applies per request and is remembered:

  • Auto — policy decides per subtask from its capabilities.
  • Local — every subtask runs on the local model. No evaluator, no cloud escalation.
  • Cloud — every subtask prefers cloud. Privacy rules still win, and if cloud is disabled it degrades to local with a note in the trace.

Privacy mode

ModeEffect
Balanced (default)High-sensitivity subtasks and prompts mentioning secrets stay local.
Strict localMedium-sensitivity subtasks stay local too.
Permissive cloudOnly explicit secret keywords force local.

Thresholds

  • Confidence (0–1) — for simple local subtasks, a local evaluator scores the answer; below this threshold Relay retries on cloud. Your 👍/👎 feedback shifts this automatically (see below).
  • Max subtasks — the cap per request; independent subtasks run in parallel up to this limit.
  • Timeout — per model call, in seconds.
Difficult work (coding, math, proofs, architecture) never runs locally while cloud is on — unless you force Local with the Route pill.

Web search

The Web pill next to the composer (or Settings → Web search) enables Ollama's hosted search API. When on, a subtask gets fresh results injected into the model prompt before it answers if it is tagged current_info or sources, or if its text mentions current-info keywords (latest, today, news, pricing, …). This works on every route — Auto, Local, and Cloud — so even a local model can answer questions about current events. The trace reason shows "Web search: N result(s)" when a search ran. The top hit is also fetched in full (via Ollama's web_fetch) so the model sees the actual page, not just snippets.

  1. Create a free key at ollama.com/settings/keys.
  2. Paste it in Settings → Web search and save, then click Test search.
  3. Toggle the Web pill on. Source links appear per subtask in the routing trace.

A failed or missing search never blocks a request — the subtask just runs without fresh context.

Answers & feedback

Every finished answer carries a privacy badge with one of three states:

  • Local only — never left your machine — every model call ran on this machine.
  • Amber "Via Ollama Cloud — ran on ollama.com" — the route stayed local, but the local model has the -cloud tag and executed on Ollama's servers.
  • Blue "N of M subtasks via cloud" — some subtasks went to cloud models through your cloud provider.
  • 👍 / 👎 — feedback tunes routing: a downvote on a fully-local answer raises the confidence threshold (escalate to cloud sooner, +0.03); an upvote lowers it (trust local more, −0.02), capped at ±0.15. Votes on cloud-routed answers are logged but don't shift the threshold.
  • ↻ Regenerate — re-runs the latest user message and replaces the latest answer (older answers can't be regenerated in place).
  • Edit and resend — hover one of your messages and click the pencil to load its text back into the composer; sending creates a new exchange.
  • Stop — while a request runs, the send button becomes a red stop square. Whatever already streamed in is kept and saved.
  • Copy — every code block has a Copy button with syntax-highlighted code; Copy answer grabs the whole response.

Secret redaction before cloud

With Redact secrets before cloud on (Settings → Routing, default on), Relay masks API keys, emails, SSNs, card numbers, private-key blocks, and password=… values with placeholders like [REDACTED_API_KEY_1] before the prompt leaves for a cloud model, then restores the originals in the answer locally. Local calls are never redacted. The trace reason shows "Redacted N secrets before cloud" when it fired.

Attachments

The paperclip accepts up to 8 files, 5 MB each: images (PNG, JPEG, GIF, WebP) and text/code files (.py, .md, .csv, .json, …). Text files are inlined into planning and the relevant subtasks. Images require a vision model — local (e.g. llava) or a cloud vision model from the pool. Attachments are saved with chat history (image previews up to ~400 KB; larger images keep a name-only chip), so they reappear when you reopen a chat.

The routing trace

Sidebar → Trace. For the latest run you get: each subtask's route and model, the why (the policy reason), duration, token usage and estimated cost, any errors, web-search source links, and the raw JSON of the whole trace. Failed subtasks get a Retry subtask button that re-runs just that subtask as a new exchange. Traces are saved with chat history, so reopening an old chat restores its last trace.

API & CLI

The UI server doubles as an OpenAI-compatible endpoint, so any OpenAI client can use Relay's full plan-route-synthesize pipeline:

curl http://127.0.0.1:8090/v1/chat/completions \
  -H "content-type: application/json" \
  -d '{"model": "relay", "stream": true,
       "messages": [{"role": "user", "content": "hello"}]}'

The model parameter can pin the route for the whole request: relay:local forces local, relay:cloud prefers cloud, plain relay uses auto routing. All three appear in GET /v1/models.

Headless CLI from the project folder:

relay ask "Summarize this repo"        # run a prompt
relay ask --stream "…"                # stream tokens
relay ask --json "…"                  # full trace as JSON
relay plan "…"                        # preview the plan without running
